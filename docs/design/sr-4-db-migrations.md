# SR-4: Plan for a Real DB Migration Path

**Status:** Design — not yet implemented  
**Pre-req:** ME-3 (memory.py → memory/ repository package) — already landed  
**Effort estimate:** Design 2–3 days; implementation 1 week

---

## 1. Decision

Adopt a lightweight numbered SQL migration framework stored inside `data/store.db`. The framework introduces a single `schema_version` table with one row; at boot, a runner inside `memory/_migrations.py` reads the current version, discovers migration files under `src/wabot_agent/memory/migrations/`, and applies any whose sequence number exceeds the stored value — each in its own transaction, updating the `schema_version` row on success. Migration files are named `0001_init.sql`, `0002_<description>.sql`, and so on; the runner sorts them lexicographically, so the 4-digit zero-padded prefix is the ordering contract. The existing `ensure_column()` helper keeps its current semantics for the duration of the transition and continues to run on every boot alongside the new runner — idempotently, as it does today. No new runtime dependency is required; the framework is approximately 150 lines of Python using only `sqlite3` from the standard library.

---

## 2. Why Not Alembic

Alembic is the correct answer for a Django- or SQLAlchemy-backed service with multiple contributors, multiple environments, and a need for autogenerated diffs against model metadata. It ships autogenerate, connection URL parsing, branching, revision heads, and Python-callable `upgrade()`/`downgrade()` functions. None of that maps onto this project. The DB is a single SQLite file on a single VPS, operated by one person, opened through a thread-local `sqlite3.connect()` call in `memory.py` lines 74–84. There is no ORM; there is no model metadata for Alembic to reflect against. Introducing Alembic means pulling in `SQLAlchemy`, adding an `alembic.ini`, wiring an `env.py`, and learning its migration-generation workflow — for a deployment surface that is one file and one process. The lightweight approach described here is easier to audit, easier to test, and easier to explain in a runbook. The one thing Alembic genuinely adds — downgrade paths — is explicitly out of scope for this project (see section 11).

## 3. The `schema_version` Table

The table holds exactly one row and one column:

```sql
CREATE TABLE IF NOT EXISTS schema_version (
    current_version INTEGER NOT NULL
);
```

On first creation, the runner inserts `current_version = 0` (or `1`, depending on the detection branch described in section 5). The runner pseudocode:

```python
def run_migrations(conn: sqlite3.Connection, migrations_dir: Path) -> None:
    conn.execute("""
        CREATE TABLE IF NOT EXISTS schema_version (
            current_version INTEGER NOT NULL
        )
    """)
    row = conn.execute("SELECT current_version FROM schema_version").fetchone()
    current = row[0] if row else 0

    files = sorted(migrations_dir.glob("[0-9][0-9][0-9][0-9]_*.sql"))
    for path in files:
        version = int(path.name[:4])
        if version <= current:
            continue
        sql = path.read_text()
        try:
            conn.executescript(sql)          # runs in its own implicit transaction
            conn.execute(
                "UPDATE schema_version SET current_version = ?", (version,)
            )
            conn.commit()
        except Exception as exc:
            conn.rollback()
            raise RuntimeError(
                f"Migration {path.name} failed — DB is at version {current}. "
                f"Fix the SQL and redeploy."
            ) from exc
```

`executescript()` commits any pending transaction before executing, so each migration file is effectively atomic at the SQLite level. The `UPDATE schema_version` and `commit()` that follow are the confirmation step; if the process dies between `executescript` and the update, the file will be re-applied on the next boot — which is why migration SQL must be written idempotently where possible (see section 9).

## 4. Directory Layout

```
src/wabot_agent/memory/
  _migrations.py          ← runner lives here (alongside _db.py post-ME-3)
  migrations/
    0001_init.sql         ← current schema captured verbatim
    0002_add_outbound_dlq.sql
    0003_reindex_inbound_by_chat.sql
```

The three example files illustrate the range: `0001_init.sql` is a snapshot of the schema as it exists when this framework ships; `0002_add_outbound_dlq.sql` represents a future additive migration (new table for tracking failed outbound sends); `0003_reindex_inbound_by_chat.sql` represents a non-additive structural change (dropping and recreating an index with a different definition). The strict naming convention is 4-digit zero-padded numeric prefix, underscore, lowercase snake-case description, `.sql` extension. The runner sorts lexicographically — `0009` before `0010`, never `9` before `10` — so the leading zeros are load-bearing. Gaps in the sequence (e.g., `0001`, `0003`, missing `0002`) cause the runner to fail at boot with a clear gap-detection error; gaps are forbidden.

## 5. Compatibility Plan: Existing `store.db` Files

This is the hardest part of the migration of the migration system. Operators running today have a `store.db` with the full current schema and no `schema_version` table. Applying `0001_init.sql` against such a file would fail immediately on the first `CREATE TABLE` — the tables already exist — or silently succeed with `IF NOT EXISTS` but leave the version row in an ambiguous state.

The runner uses a two-branch boot sequence:

- **No `schema_version` table present** (existing operator DB): assume the DB is at version 1. Run `ensure_column()` for any forward-compatibility patches that the existing `_init_db()` already applies. Then create the `schema_version` table and insert `current_version = 1`. Do not apply `0001_init.sql`. On subsequent boots, the standard path applies.
- **`schema_version` table present**: read `current_version`, apply any `NNNN_*.sql` with a higher sequence number in order.

Detection sketch:

```python
row = conn.execute(
    "SELECT name FROM sqlite_master "
    "WHERE type='table' AND name='schema_version'"
).fetchone()
if row is None:
    # existing store.db — bootstrap without re-running init
    _bootstrap_existing_db(conn)   # ensure_column calls + insert version 1
else:
    run_migrations(conn, migrations_dir)
```

The detection query is the canonical SQLite introspection idiom; it reads the schema catalog directly and does not depend on any application table existing. After `_bootstrap_existing_db()` runs once, the `schema_version` row exists, and every subsequent boot takes the standard path. The one-time bootstrap is idempotent: if it is interrupted mid-way (e.g., the process is killed), the absence of the `schema_version` table will cause it to run again on the next boot — which is safe because `ensure_column()` is a no-op when the column already exists.

## 6. What `0001_init.sql` Contains

`0001_init.sql` is a verbatim snapshot of the current schema, not a derivation from the Python code. The capture process is:

```
sqlite3 data/store.db .schema > src/wabot_agent/memory/migrations/0001_init.sql
```

Then append the `schema_version` DDL at the bottom of that file (with `CREATE TABLE IF NOT EXISTS schema_version ...`). The resulting file is approximately 120 lines: the 10 `CREATE TABLE` blocks currently defined in `memory.py` lines 102–203, the 3 `CREATE INDEX` statements for `scheduled_reminders`, `outbound_tasks`, and `web_research_jobs`, and the `schema_version` table itself. The file is never regenerated from code; it is a fixed artifact committed once. Future schema changes go in `0002_*.sql`, `0003_*.sql`, and so on. The only reason to touch `0001_init.sql` after it is committed is to fix a syntactic error discovered before any operator has run the framework — once it is in production, it is immutable.

## 7. Coexistence with `ensure_column`

`ensure_column()` (currently at `memory.py` line 214) keeps its existing semantics for the duration of the transition. When called during boot, it reads `pragma table_info(<table>)` and issues `ALTER TABLE ... ADD COLUMN` if the column is absent. Seven calls currently live in `_init_db()`: three for `processed_messages` (`status`, `run_id`, `error`) and four for `inbound_messages` (`media_kind`, `media_mime`, `media_filename`, `has_media`). These calls run on every boot and are fully idempotent — the column either exists or gets added; no error either way.

New columns that arrive after SR-4 lands should come via a numbered `NNNN_*.sql` migration, not via a new `ensure_column()` call. The `ensure_column` back-compat path exists specifically for the seven existing callers, which represent columns that were added to tables after those tables were first created — the pattern that spawned this whole ticket. The plan is to consolidate these seven calls into a single follow-up PR that converts each one to an idempotent `ALTER TABLE ... ADD COLUMN` inside an appropriately numbered migration file and removes the `ensure_column()` calls from `_init_db()`. Until that PR lands, both mechanisms run on every boot: the migration runner first, then the `ensure_column()` calls. The double-boot pattern is safe because both sides are idempotent, and the `ensure_column()` path is a strict subset of what the migration files will do once the consolidation PR ships.

## 8. Failure Modes

**Migration fails partway through.** `executescript()` in SQLite commits before executing, so partial execution of a multi-statement migration is possible if the script has no explicit `BEGIN`/`COMMIT` wrapping. The runner calls `conn.rollback()` on exception and raises a `RuntimeError` that names the failing migration file and the version the DB is still at. Boot fails loudly; the FastAPI process does not start. The operator must fix the SQL (or restore from backup), then redeploy. The `schema_version` row retains its pre-migration value — the next boot retries the same file.

**Migration file is missing (gap in sequence).** The runner checks that the sequence from 1 to the highest numbered file contains no gaps before applying anything. If file `0003_*.sql` exists but `0002_*.sql` does not, boot fails with a gap-detection error listing the missing numbers. This prevents silent version skips that would leave the DB in an undefined intermediate state.

**Migration file is modified after being applied.** There is no content hash check. The runner trusts the sequence number; it does not re-verify previously applied files. This is explicitly out of scope. Operators are warned in the runbook: do not edit committed migration files. If a migration is discovered to have been applied incorrectly, the correct path is a new numbered migration file that corrects the state, not editing the old one.

## 9. Operations

**Writing a new migration.** Add a file named `NNNN_<description>.sql` where `NNNN` is the next integer in sequence after the current highest. Write SQL that is idempotent where SQLite allows: use `CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, `DROP INDEX IF EXISTS`. For `ALTER TABLE ... ADD COLUMN`, SQLite does not support `IF NOT EXISTS` in the column-add form — write the migration knowing it will only ever run on a DB at the prior version. Test against a copy of `data/store.db` from production before deploying: `cp data/store.db /tmp/test.db && sqlite3 /tmp/test.db < src/wabot_agent/memory/migrations/0005_new_thing.sql`. Commit the file; CI will pick it up; the runner applies it on next boot.

**Rolling back.** There is no automated rollback. Migrations are up-only by design. If a migration must be undone, the process is: stop the service, open a `sqlite3 data/store.db` session, manually execute the inverse SQL (e.g., `DROP TABLE`, `DROP INDEX`, `DROP COLUMN` if SQLite version supports it), update `schema_version` to the prior version number, and restart. A known-good backup taken before the migration is the safety net. The runbook should specify that operators take a backup before every deploy that includes a new migration file: `cp data/store.db data/store.db.bak.$(date +%Y%m%d%H%M%S)`.

**Inspecting migration status.** The current applied version is always readable without opening Python:

```
sqlite3 data/store.db "SELECT current_version FROM schema_version"
```

Compare the output against the highest `NNNN` file in `migrations/` to determine whether any migrations are pending. A discrepancy means either a deploy is in progress or something went wrong on the last boot.

## 10. Pre-requisites

ME-3 (memory.py → memory/ repository package) must be landed before implementing SR-4. The reason stated in the MASTER doc is exactly right: the migrations runner should live in `memory/_migrations.py` alongside `_db.py`, not bolted onto a 1,100-line god module. With ME-3 done, the package layout provides a natural home for both the runner and the `migrations/` subdirectory, and the `_db.py` connection factory (`get_thread_connection`, with its WAL pragma, `check_same_thread=False`, and `row_factory`) is already isolated in one place. The runner will call through `_db.py` rather than opening its own connection, keeping the WAL pragma and busy-timeout in one authoritative location.

## 11. Out of Scope

The following are explicitly not part of this design or the implementing PR:

- No code changes in this document — SR-4 is a design doc only; the implementing PR is a separate ticket.
- No rename of `store.db`. The file name is referenced in `.gitignore`, `config.py`, tests, and the CLAUDE.md safety rules; renaming it is a separate migration with its own risk surface.
- No introduction of Alembic or any ORM-driven migration tooling.
- No Python-callable migrations. Data backfills that require Python logic (e.g., computing a hash of existing rows) will go in a `0NNN_<name>.py` mechanism to be designed separately if and when needed; they are not addressed here.
- No automatic backups. Taking a backup before deploying is operator responsibility; the runbook documents the `cp` command, but the application does not automate it.
- No online schema changes or rolling deploys. The application is a single-process, single-VPS deployment; there is no need for zero-downtime schema evolution, and SQLite's WAL mode already serializes writes without a separate migration lock.
- No hash verification of applied migration files. Content integrity is a runbook concern, not an automated check.
